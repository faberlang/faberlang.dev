Features

The architectural ideas that make Faber distinctive

Faber is not a conventional programming language. Its design is shaped by a small set of architectural commitments that together produce a language unlike most mainstream alternatives. This section documents those commitments and the features that follow from them.

Reader locale

Source code and diagnostics rendered in seven human languages — Thai, Chinese, Arabic, Hindi, Vietnamese, and Latin — through the same mechanism that renders Faber to Rust or WebAssembly.

7 packs shipped · Localized keywords · Code-first diagnostics · bidi isolation

Compilation lanes

One frontend, three lowering routes: HIR-direct (application), MIR (systems), and AIR (transform detour for autodiff and fusion).

HIR · MIR · AIR · Shared frontend · Semantic ownership

Latin vocabulary and glyphs

Behavioural vocabulary in Latin (functio, si, redde), structural meaning in universal glyphs (← → ∴ ≡ ∪ ⇥). The two never trade places.

Type-first · Glyphic structure · Latin keywords · Declarative

Commandments

Nine design laws that make Faber feel like Faber: types before names, mechanical over magical, one sign one job, absence is typed, and more.

Design principles · Invariants · Language feel

Canonical vs sugar surfaces

Multiple parseable surfaces, one semantic shape. Numeric type sugar (tf32, mf32), annotation sugar, and author vs canonical formatting modes.

Numeric sugar · Annotation sugar · Format modes · Dual surface

Capability calls and frames

Structured communication between Faber programs and host runtimes through typed conversations (sermo), frame envelopes (scrinium), and capability calls (ad).

ad · sermo · scrinium · status · meus · tuus · Host gateway

Inline testing

First-class test framework built into the language with probandum, proba, and adfirma — tests live alongside source in the same file.

probandum · proba · adfirma · tag · Package-scoped tests
