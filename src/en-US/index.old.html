Faber is a package-oriented programming language with a Latin behavioral vocabulary and a small, regular grammar. Source lowers through the Radix compiler to reviewable Rust and native binaries. Its defining architectural property is that meaning lives in the semantic core, not in any rendering — neither the Latin you read nor the Rust it emits is the real program. Both are renderings of the same intermediate representation. The application lane ships packages and CLI tools through Rust; the systems lane lowers through MIR for tensors, GPU kernels, and low-level staging.

The name means maker or craftsman. Its compiler is named Radix — root. Status: experimental, pre-1.0.

1. Why Faber

Faber is a readable semantic staging language. You write program intent in a small, regular source language, then lower it through one of two lanes: the application lane (HIR → Rust packages and native binaries) or the systems lane (MIR → FMIR packages, GPU text, validation artifacts, and low-level staging). Faber's own grammar is the centre; target backends exist to ship, test, and inspect that source.

The guiding question for Faber source:

Can a reader see the semantic role of every construct before knowing the target backend?

Faber stands on three deliberate signals that work together:

  • Type-first declarations — shape reads toward binding: textus nomen, not nomen: textus.
  • Latin behavioural words — declarations, statements, lifecycle, and intent: functio, genus, fixum, redde, si.
  • Structural glyphs — value flow, type flow, and structural joints: ←, →, ∴, ≡, ∪.

The net effect: source with stable grammatical shape before it lowers to any backend. Its value is strongest when the source needs to be reviewed, transformed, and lowered without losing the reader's sense of intent.

2. Commandments

These are the rules that make Faber feel like Faber. Syntax can evolve, but changes should preserve these laws.

  1. Types before names. Declarations read from shape to binding: textus nomen, numerus aetas, functio salve(textus name) → textus.
  2. Mechanical over magical. The same construct should mean the same thing everywhere. If a reader needs distant context to know what a symbol does, the syntax is suspect.
  3. Glyphs carry structure. Use glyphs for operators, control-flow joints, and value-flow edges: ←, →, ⇥, ∴, ≡, ∪.
  4. Latin carries behaviour. Use Latin words for declarations, statements, lifecycle, and behavioural intent: functio, genus, fixum, varia, redde, cape.
  5. Conjugation carries time and flow. When the same root logic can run synchronously, asynchronously, or as a generator, the conjugated form should carry that execution mode.
  6. One sign, one job. A glyph or keyword may have exact aliases, but it should not carry unrelated meanings.
  7. Runtime flow is explicit. Runtime binding, reassignment, and mutation use ←; structural definition uses =.
  8. Absence is typed. Nullable value types are written as unions, such as T ∪ nihil; optional declaration slots use post-name markers such as sponte.
  9. The compiler does not guess to hide missing information. Missing type information is an analysis problem to fix upstream, not a codegen detail to paper over.

3. Language Snapshot

Faber is type-first and glyph-forward:

functio divide(numerus a, numerus b) → numerus ∪ nihil {
    si b ≡ 0 ∴ redde nihil
    redde a / b
}

functio quotient(numerus a, numerus b) → numerus ⇥ textus {
    si b ≡ 0 ∴ iace "division by zero"
    redde a / b
}

genus Point {
    numerus x
    numerus y
}

incipit {
    fixum _ p ← Point {
        x = 10,
        y = 20
    }

    nota "Salve, §!"(p.x)
}

The fastest way to recognise Faber:

  • Types come before names: textus nomen, not nomen: textus.
  • Runtime binding and reassignment use ←; = is for structural field shape only.
  • Function returns use →.
  • Recoverable alternate exits use ⇥; iace throws on that channel and fac / cape catches it.
  • Compact branch bodies use ∴ or accepted alias ergo.
  • Nullable values use T ∪ nihil.
  • Latin words carry declarations, statements, lifecycle, and behaviour.
  • Glyphs carry value flow, type flow, and structural joints.
  • Formatted textus uses string-template application: "…"(args…) with § holes.

5. Two compilation lanes

Faber is one language with two compilation lanes. They share the same frontend (parser, typechecker, exempla) but differ in what they optimise for and how you ship the result.

Source → Lex → Parse → Collect → Resolve → Lower → Typecheck → Analysis │ ┌─────────────────────────┴─────────────────────────┐ │ │ HIR backends MIR backends │ │ Rust · Faber · TypeScript · Go fmir · wasm · llvm-text · metal-text · wgsl-text · sexp

Application lane (primary product)

Write Faber packages with faber.toml, import Latin libraries through flat norma:* modules, and ship reviewable Rust that Cargo links into a native binary. This is the path for CLI tools, OS-level programs, API servers, and script-style hosts.

DeliverableBackendPackage build / run
Rust binaryHIR → codegen/rustYes — primary
TypeScript filesHIR → codegen/tsNo — file emission
Go filesHIR → codegen/goNo — file emission
Canonical Faber sourceforma formatterNo — use faber format
 TypeScript and Go help prove Faber's semantics across target shapes; Rust is the only supported way to build and run Faber applications today.

Systems lane (advanced compilation)

For tensors, GPU kernels, and low-level staging, Faber lowers through MIR. fmir-text, fmir, and fmir-bin are executable MIR package/runtime targets. Systems MIR text/binary emitters (wasm*, llvm-text, metal-text, wgsl-text, sexp) are supported-with-limitations targets with fail-closed subsets.

In-process interpretation is faber script (MIR stepper). The systems lane is where systems-level engineering happens — not the default HIR → Rust package path.

Rust is the behavioural reference. When a systems target can name a tensor runtime symbol but not execute tensor semantics, that is recorded as import/probe-only — the target can print the shape, not run it.

6. Quick start

Install the released CLI from the Homebrew tap:

brew install ianzepp/tap/faber
faber --version

Naming note: the faber crate name on crates.io is a different project. This project ships via the Homebrew tap above or by building from source.

From a checkout, the fastest path to "it works":

./scripta/ci                                         # release validation gate
cargo run --manifest-path ../faber/Cargo.toml -- check ../examples/corpus/incipit/salve-munde.fab
cargo run --manifest-path ../faber/Cargo.toml -- build ../examples/corpus/incipit/salve-munde.fab
cargo run --manifest-path ../faber/Cargo.toml -- explain ≡  # glyph reference, disk-backed
cargo run --manifest-path ../faber/Cargo.toml -- targets       # live backend capability flags

Create a starter package:

cargo run --manifest-path ../faber/Cargo.toml -- init hello

The disk-backed reference pack covers the current package and test contracts:

cargo run --manifest-path ../faber/Cargo.toml -- explain manifest
cargo run --manifest-path ../faber/Cargo.toml -- explain proba
cargo run --manifest-path ../faber/Cargo.toml -- explain probandum

Package manifest

Faber packages are the application lane unit of delivery. A faber.toml manifest describes project metadata and Rust build configuration:

[package]
name = "salve"
version = "0.1.0"
edition = "2026"

[paths]

[build]

kind = "bin"

7. CLI roles

Faber is the user-facing project and package tool:

  • faber check, faber build, faber targets
  • faber explain <term>, faber explain --search <query>
  • faber init
  • faber run; faber test
  • faber format
  • Compatibility aliases: faber lex, faber parse, faber hir, faber emit, faber cli-ir

Radix is the compiler-developer tool:

  • radix lex, radix parse, radix hir, radix mir, radix emit, radix check, radix targets

Package policy lives in Faber; use faber check <package-dir> and faber build for local packages.

Compiler performance

Radix's frontend scales roughly linearly with source size at ~16 MB/s, ~60 ns per byte, in-process and single-threaded:

500 functions / ~3.3K lines~52 KB~3 ms

 Release build, target=rust, full pipeline including codegen. The largest real example today is ~140 lines.

For the Rust target, faber build emits a Rust crate and hands it to Cargo. Faber's dependency model is designed so the heavy parts compile once and are reused:

8. Standard library

On the application lane, Faber programs import Latin library and data modules through flat norma:* paths:

  • Filesystem: norma:solum, norma:solum/path
  • HTTP: norma:http
  • Processes: norma:processus
  • Data formats: norma:json, norma:toml, norma:yaml
  • Tensor helpers: norma:tensor
  • Codec navigation: norma:valor

Stdlib declarations are Faber source under the sibling norma repository. Native Faber bodies are the default implementation tier; effectful modules open ad route conversations, and behaviour not yet expressible natively carries an explicit mori deferred body. 

functio leget(textus via) → textus {
    redde ad 'solum:leget' (via) ↦ textus
}

Graphics and geometry: the optional triga:* library provides geometry, scene, and GPU-facing type contracts. Resolved through FABER_LIBRARY_HOME and installable as a Cista source package.

9. Reader locale and diagnostics

Reader-language packs are a first-class compiler surface. A pack maps Faber keywords, primitive type spellings, and code-first diagnostics into a reader locale. Seven packs ship today: la, ar, hi, vi, th-TH, zh-Hans, zh-Hant.

la is the default reader surface — Latin Faber keywords and canonical type spellings mapped to themselves, with English diagnostic message templates. Other locales remap keyword/type spelling and render the same structured diagnostics in the reader language.

Diagnostics are structured facts before prose: each carries a stable code (LEX###, READER###, PARSE###, SEM###, WARN###) and named arguments; the pack owns the rendered template text.

Select a locale with the --reader-locale <locale> flag or the faber.toml [reader] table:

[reader]
locale = "zh-Hans"

The compiler validates installed packs against the generated Latin scaffold. See radix::reader_locale for the runtime API.

The reference set is chosen for architectural stress

Five reference locales, picked so that collectively they force every Unicode and emission problem the substrate must survive:

RenderingAccessReachUnique architectural stress
th-TH · ไทยHighMediumSpaceless script; the tokenizer stress test.
zh-Hans / zh-HantMediumVery highPaired keywords; sibling packs with inheritance.
ar · العربيةHighHighRight-to-left and bidi; contextual shaping.
hi · हिन्दीHighVery highMatra/virama clusters; Indic numerals.
vi · Tiếng ViệtMediumHighHeavy diacritics on Latin script; NFKC edge cases.

This section is a practical orientation to Faber source. It occasionally uses other languages as contrast, but Faber's grammar is the authority; it is not a variant spelling of another language.

Borrowing and mutability (de, in, ex)

Faber marks how a value is passed with short prepositions on parameters:

    nota label
}

functio duplica(in numerus value) → vacuum {
    value ← value * 2
}

functio consume(ex textus buffer) → textus {
    redde buffer
}

functio salve(textus nomen) → textus {
    redde "Salve, §!"(nomen)
}

Same words, different jobs

Structural declarations

    numerus x
    numerus y
}

    x = 10,
    y = 20,
}

Control flow shape

// values
itera ex items fixum item {
    si item ≡ target ∴ redde item
}

// keys
itera de table fixum key { ... }

// range
itera ab 0‥10 fixum i { ... }

// compact branch
si score ≥ 90 ∴ redde "A"
sin score ≥ 80 ∴ redde "B"
secus ∴ redde "C"

// guard section
custodi {
    si b ≡ 0 {
        redde 0
    }
}

Canonical surface

When generating or reviewing Faber, default to these forms:

fixum textus name ← "Marcus"
varia numerus count ← 0
functio greet(textus name) → textus { ... }
functio parse(textus raw) → numerus ⇥ textus { ... }
si ready ∴ redde value
si invalid ∴ iace "bad input"
fixum _ n ← "42" ↦ numerus ⇥ 0
fixum _ x ← value ∷ textus
functio maybe() → textus ∪ nihil

Error handling:

fac {
    iace "bad input"
}
cape err {
    nota err
}

11. Core type surfaces

Type and size generics

Functions, type aliases, genus, and implendum declarations accept type parameters with <T> syntax:

functio identitas<T>(T valor) → T {
    redde valor
}

fixum _ value ← identitas<numerus>(7)

Numeric widths

numerus and fractus have default widths (i64 and f64) and explicit width forms:

fixum numerus<i32> narrow ← 7 ∷ numerus<i32>
fixum numerus<u64> wide ← 255 ∷ numerus<u64>
fixum fractus<f32> single ← 1.5 ∷ fractus<f32>

Tensors and sparsa

tensor<T, Figura> is the dense form: every coordinate in the static shape has a stored value. sparsa<T, Figura> is the sparse sibling. Both use the same Figura discipline. 

fixum tensor<fractus<f32>, []> scalar ← vacua
fixum tensor<numerus, [4]> vector ← [1, 2, 3, 4] ↦ tensor<numerus, [4]>
fixum numerus ∪ nihil first ← vector[0]

// Sugar for numeric/tensor-heavy code:
fixum tf32[4] lanes ← [1.0, 2.0, 3.0, 4.0] ↦ tf32[4]

↦ converts between dense and sparse only when the element type and shape are the same. The t and s sugar families mirror each other: tf32[2, 3] means tensor<f32, [2, 3]>, sf32[2, 3] means sparsa<f32, [2, 3]>.

GPU core types

The systems lane recognises GPU-facing register and storage types: f16, matrix<T, [R, C]> (register-class value type), and atomic<T> (storage-sensitive cell type for i32 and u32). These types are intentionally target-gated: targets that lack a real carrier fail closed.

12. Return and error channels

Faber separates three related ideas that many languages collapse into one shape:

  • → T declares the normal success return channel.
  • T ∪ nihil makes absence part of the success value domain.
  • ⇥ E declares a recoverable alternate-exit channel for errors.
// Normal return
functio porta(numerus x) → numerus {
    si x < 0 ∴ redde 0
    redde x * 2
}

// Alternate exit channel
functio divide(numerus a, numerus b) → numerus ⇥ textus {
    si b ≡ 0 ∴ iace "division by zero"
    redde a / b
}

// Local recovery with fac / cape
functio tutum(numerus a, numerus b) → numerus {
    fac {
        redde divide(a, b)
    }
    cape err {
        mone err
        redde 0
    }
}

// Effect-only failable
functio exigePositivum(numerus value) ⇥ textus {
    si value < 0 ∴ iace "negative value"
}

13. Runtime binding (←) vs structural definition (=)

GlyphRoleUse for
←Runtime flowInitial binding, reassignment, and mutation at execution time
=Structural shapeField names inside literals and declaration metadata — not runtime stores
 
fixum numerus count ← 0
varia textus label ← "ready"
count ← count + 1

genus Point {
    numerus x
    numerus y
}

fixum _ p ← Point {
    x = 10,
    y = 20,
}

fixum bindings are write-once: you may declare without an initializer and assign exactly once later, or bind immediately. A second assignment to the same fixum local is rejected. varia bindings remain freely reassignable.

§ (U+00A7) is the template hole in Unicode textus/forma forms only. String-template application uses "…"(args…):

}

// Captured templates (forma) for bound SQL/URL payloads:
fixum _ query ← `select * from users where id = §`(user_id)

15. Inline JSON (json)

A bare { … } is inline JSON: a compile-time json document, not an anonymous Faber object. Keys are quoted JSON strings separated by :; the root must be an object. 

fixum _ empty ← {}
fixum _ user ← { "name": "Marcus", "age": 30, "active": true }
fixum _ nested ← { "meta": { "version": 1 }, "tags": ["alpha", "beta"] }

For typed genus construction with Faber expressions, use the type name and = field shape — not JSON : syntax: 

fixum _ p ← Point { x = 10, y = 20 }

17. Nullability and optionality

Use T ∪ nihil when the value can be absent. Use sponte after the name when a parameter or field may be omitted by the caller or constructor. 

functio find(textus key) → numerus ∪ nihil

functio connect(textus host, numerus port sponte) → vacuum

genus User {
    textus email sponte
}

18. Conversion and construction

Two important conversion operators:

fixum _ parsed ← "42" ↦ numerus
fixum _ safe ← input ↦ numerus ⇥ 0
fixum _ text ← value ∷ textus
  • Use ↦ for runtime conversion, especially parsing or coercion that may fail.
  • Use ∷ for explicit static type ascription (postfix, target-type driven).
  • Use Type { field = value } for records.
  • Use value ↦ Type for type-directed materialization.

21. Status

Faber 1.0.0 freezes the grammar for the major line. The compiler, Radix, is private; its output is not — the grammar, runtime, examples, tree-sitter package, and every lowered artifact are public and reviewable. The staging language exists so that the thing you review is the thing that runs.

Reader locale is proposed. It is a north-star design with real substrate prerequisites and an open pack-delivery architecture. This page renders illustrative pack spellings taken from the design document; no pack has been authored.
