# Goal: Presentable code-sample tabs — animated locale/example blocks

## Summary

Add one shared, progressively-enhanced code-block component to faberlang.dev —
a tabbed card with window chrome, styled entirely from `speculum.css` tokens —
used in **two configurations** on the site's two highest-visibility surfaces:

1. **Portal hero (locale axis):** on the root language portal (`/`), below the
   language-chooser globe, present *one* `salve-munde` program in every
   reader-locale as a single animated tabbed card that **types itself once** on
   scroll-into-view. This makes the compiler's thesis — one semantic core, many
   renderings — the first thing a visitor sees.
2. **Landing tabs (example axis):** on each per-language landing page
   (e.g. `/en-US/`), present *several different short programs*, all rendered in
   **that page's** reader-locale, as a tabbed card (no typing). Once a visitor
   has chosen their language, the useful question is "what does Faber look like
   across a few examples in my own tongue," not "what is this in Thai."

It is **one component, two data sources.** The tab strip does not care whether
tabs represent locales or examples.

This is deliberately **deferred** — captured now, built after the current
multilingual translation work settles. No generator changes are made by writing
this doc.

## Problem

- The single most compelling property of Faber — the same program rendered in
  Thai / Chinese / Arabic / Hindi / Vietnamese / Latin keywords — is currently
  shown as a **static grid** of `.porta-sample` cards on the portal
  (`#porta-samples`, from `generate-portal.py`). It reads as a reference table,
  not as the "wow" moment it actually is.
- Every code example across the docs is a generic monochrome `pre.faber-code`
  blob. There is no higher-tier presentation for the pages that most need to
  make a first impression (portal, getting-started).
- The docs site ships **zero JavaScript** today, so any richer, interactive
  presentation is a genuine (small) architectural addition, not a tweak.

## Goals

- A single component (neutral name, e.g. `.faber-demo-tabs`) with:
  - window-chrome header (file name, copy button),
  - a tab strip (labels from `locales.toml` native names, or example titles),
  - one visible panel at a time with a light crossfade,
  - **progressive enhancement**: with no JS it degrades to a labeled vertical
    stack of real `<pre>` DOM text — readable, copyable, agent/`llms.txt`-safe,
    RTL-correct.
- **Portal hero** (locale axis, with one-shot typing) replacing/upgrading the
  existing `#porta-samples` grid in `generate-portal.py`.
- **Per-language landing tabs** (example axis, no typing) emitted from the
  Markdown pipeline for `src/{locale}/index.md`.
- Shared CSS in `generator/www/speculum.css` (the one stylesheet) and a single
  small JS enhancer asset in `static/` (copied to `dist/` as-is), referenced
  from both generators so the two configurations never drift.
- Accessibility: ARIA `tablist`/`tab`/`tabpanel`, arrow-key navigation,
  visible focus, `dir="rtl"` on RTL panels, and full `prefers-reduced-motion`
  opt-out (no typing, no fade).

## Non-goals

- Applying tabs to *every* code fence. This is reserved for high-visibility
  surfaces; ordinary fences stay `pre.faber-code`.
- Real syntax highlighting / tokenized coloring. The generator emits
  `<code class="lang-…">` but does not currently paint `kw`/`typ` spans on block
  code; the component is monochrome unless/until that lands. Out of scope here.
- A client-side transcoder. Locale/example renderings are produced at build time
  by the existing `faber format --reader-locale` path; the component only
  displays pre-rendered escaped text.
- Building the root portal. It **already exists and is deployed** — this goal
  upgrades one section of it.
- Localizing portal chrome strings (portal is intentionally multi-script).

## Ground Truth Researched (2026-07-20)

- **Portal is live**, generated by `generator/scripts/generate-portal.py`
  (Python, invoked from `build-site.sh` step 8), *not* the Faber Markdown
  pipeline. `dist/index.html` = `Faber · porta`, `<body class="porta">`.
  - Layout order: `.porta-gate` → `.porta-ring` (the globe: positioned
    `.porta-node` selectors for 7 locales) → `#porta-samples` (a **static grid**
    of `.porta-sample` cards, one per locale).
  - Sample content: `load_sample()` reads
    `exemplars/salve-munde.{reader_locale}.fab`; each card is
    `<pre class="faber-code"><code{dir}>…</code></pre>`, RTL-aware.
  - Guarded by smoke tests in `build-site.sh:274-282` (asserts `class="porta"`,
    native names, `incipit`, and *no* meta-refresh).
  - **The portal hero is therefore an upgrade of `#porta-samples`, not a new
    build — the per-locale data plumbing already exists.**
- **Per-page fences** are parsed in `generator/src/markdown.fab` (~line 313),
  emitting exactly `<pre class="faber-code"><code class="lang-{info}">…</code></pre>`.
  A richer fence-info model already exists but is unused for rendering:
  `generator/src/fence.fab` `parse_info` extracts `locale=`, `pinned`/`fluid`,
  `compiles`/`rejects`/`output-matches` (used only for CI validation via
  `types.fab`).
- **HTML assembly**: `generator/src/html.fab` (~line 46) builds `<head>`/`<body>`;
  this is where a `<script>` include for the enhancer would be added.
- **No JS ships today** (no `.js` in `dist/`, no `<script>` in pages).
- **One stylesheet** rule (AGENTS.md): all CSS in `generator/www/speculum.css`;
  the generator cannot write CSS or inline styles. `static/` is copied to
  `dist/` as-is — the JS asset lives there.
- **Tab labels / native names** come from `generator/locales.toml`
  (`native_name`, `native_script`, `status`, `reader_locale`).
- Reference source for the locale-axis content already exists verbatim in
  `src/en-US/features/reader-locale.md` (the 7-locale `salve-munde` stack).

## Prototype

The approved design is [`prototype-v2.html`](./prototype-v2.html) (standalone,
self-contained; open in a browser). It demonstrates: (1) the tabbed variant,
(2) the hero variant with one-shot typing, (3) the current baseline for
comparison — all styled with `speculum.css` tokens, both light/dark, RTL
Arabic, reduced-motion opt-out. Design decisions locked 2026-07-20:

- **Component name:** `.faber-demo-tabs` / `.fdt-*` (neutral; tabs aren't
  always locales).
- **Chrome bar:** 20px `f` brand-mark + filename + copy button. No
  traffic-light dots (borrowed chrome, rejected).
- **Connected tabs:** active tab joins the panel surface (`paper-alt`), drops
  onto the strip rule, carries a 2px `--glyph` top-accent echoing the
  renderbar. Tabs flex to fill the strip evenly (`min-width: max-content`;
  whole tabs wrap, never truncate).
- **Elevation grammar:** default variant rests at `--shadow-soft`, lifts on
  hover/focus-within; `.fdt-hero` (portal) holds `--shadow-lift` permanently.
- **Content:** 7 locales, real `pack.toml` vocabulary only. Every panel in the
  prototype passes `faber check --reader-locale <locale>`; la/ar canonical
  forms verified structurally identical. (v1's invented translations were
  wrong; do not reuse them.) zh-Hant panel uses 入口 for `incipit`, ahead of
  its pack, matching the checked-in exemplar.
- **RTL:** `dir="rtl"` on the `<pre>` itself (live-portal precedent).
- **ARIA:** `aria-controls`/`aria-labelledby`, roving `tabindex`,
  Arrow/Home/End keys.
- **Panel height:** free reflow on tab switch (no min-height).
- **Open content swap:** panels still show function-`salve`; the portal hero
  upgrades to the `Exitus` sum-type program (validated, runs) once its six
  locale translations exist.

[`prototype.html`](./prototype.html) (v1) is retained for history —
superseded; its translations do not match the real packs.
Original published artifact (interactive): `claude.ai/code/artifact/e3d3223e-0a4e-435d-a949-fadfdfcf75d4`.

**Substrate blocker discovered 2026-07-20:** `faber format --reader-locale`
desugars through HIR (and packs cover only ~38/111 keywords; zh-Hant 6/111),
so build-time locale rendering is not viable for showcase surfaces. Content
must be hand-maintained per-locale exemplars (the existing
`stdlib/reader/<locale>/exemplars/` pattern), CI-gated with `faber check`.
Full report: `radix/docs/factory/reader-locale-coverage-gaps/audit.md`.

## Stage graph

```text
S1  shared surface: .faber-demo-tabs CSS (speculum.css)
    + enhancer JS (static/) + <script> include (html.fab)
        ↓
S2  Phase A — portal hero (locale axis, typing):
    upgrade #porta-samples grid → tabbed hero in generate-portal.py
        ↓
S3  Phase B — landing tabs (example axis, no typing):
    fence-group emission in markdown.fab for src/{locale}/index.md,
    examples rendered per reader_locale via faber format
        ↓
S4  full build + gates green; regenerate dist/; commit
```

**Ship Phase A before Phase B.** The portal hero is lower-lift (data exists) and
higher-impact; the landing example-axis tabs need an authoring/marker mechanism
and per-locale example rendering.

## Checkpoints and gates

**Acceptance**

- [ ] No-JS: portal and landing render a readable labeled stack of real `<pre>`
      text (view-source / JS-disabled check).
- [ ] Portal hero types the canonical sample once on scroll-in, then tabs stay
      live; grid content parity with prior `#porta-samples`.
- [ ] Landing tabs show ≥3 distinct examples, all in the page's reader-locale.
- [ ] RTL panels (`ar`) render `dir="rtl"`; copy copies the active panel.
- [ ] Keyboard: tab focus + arrow-key switching; visible focus ring.
- [ ] `prefers-reduced-motion`: no typing, no fade.
- [ ] `build-site.sh` green; existing portal smokes still pass.

**Validation**

```bash
bash generator/scripts/build-site.sh
rg -n 'faber-demo-tabs|data-typing' dist/index.html dist/en-US/index.html
! rg -q 'http-equiv="refresh"' dist/index.html   # portal must not refresh
```

**Release:** defer push without operator authorization (house rule).
