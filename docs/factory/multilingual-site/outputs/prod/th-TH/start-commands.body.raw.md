นี่คือแผนผังคำสั่ง CLI สำหรับสัปดาห์แรกของการทำงานกับ Faber ใช้หน้านี้เป็นดัชนีคำสั่ง แล้วเปิดหน้า [เครื่องมือ build ของ Faber](/tooling/faber-build-tool.html) โดยละเอียดเมื่อต้องการดูแฟล็กและรายละเอียดของไปป์ไลน์คอมไพเลอร์

## วงจรประจำวัน {#daily-loop}

| คำสั่ง | ใช้สำหรับ |
|---|---|
| `faber check <package>` | ตรวจสอบส่วนหน้าของแพ็กเกจอย่างรวดเร็ว: วิเคราะห์โทเค็น แยกวิเคราะห์ ตรวจชนิด และลดรูป |
| `faber build <package> -t rust` | สร้างโปรเจกต์ Rust สำหรับตรวจทานหรือคอมไพล์เป็นไบนารีเนทีฟ |
| `faber run <package>` | สร้างและเรียกใช้แพ็กเกจแอปพลิเคชัน |
| `faber test <package>` | เรียกใช้การทดสอบของแพ็กเกจเมื่อแพ็กเกจกำหนดพื้นผิวการทดสอบไว้ |
| `faber explain <code>` | อ่านคำอธิบายการวินิจฉัยที่มีรูปแบบคงที่ |

เริ่มด้วย `check` คำสั่งนี้มีต้นทุนต่ำที่สุด และเป็นคำสั่งที่เอเจนต์ควรเรียกใช้ก่อนเสนอว่าโค้ดที่สร้างขึ้นนั้นเป็น Faber ที่ถูกต้อง

## Check {#check}

<<<FENCE 0>>>

ผลการตรวจสอบที่ผ่านหมายความว่าแพ็กเกจถูกต้องตามไวยากรณ์และความหมายสำหรับส่วนหน้าของคอมไพเลอร์ แต่ไม่ได้หมายความว่าได้เรียกใช้ชุดเครื่องมือเนทีฟขั้นสุดท้ายแล้ว

## Build {#build}

<<<FENCE 1>>>

เป้าหมาย Rust ออกแบบมาให้ตรวจทานได้โดยเจตนา Rust ที่สร้างขึ้นเป็นอาร์ติแฟกต์ของคอมไพเลอร์ ไม่ใช่แหล่งความจริง ให้แก้ไขแพ็กเกจ Faber แล้ว build ใหม่ แทนการแก้ไข Rust ที่สร้างขึ้นด้วยมือ

## Run {#run}

<<<FENCE 2>>>

ใช้ `run` กับแพ็กเกจแอปพลิเคชันที่มีจุดเริ่มต้น `incipit` ส่วนแพ็กเกจที่มีเฉพาะไลบรารีควรใช้ `check` และ `test` แทน

## อธิบายการวินิจฉัย {#explain}

<<<FENCE 3>>>

กลุ่มการวินิจฉัยเป็นตัวอ้างอิงที่มีรูปแบบคงที่: `LEX` สำหรับข้อผิดพลาดด้านการวิเคราะห์โทเค็น `PAR` สำหรับข้อผิดพลาดของตัวแยกวิเคราะห์ และ `SEM` สำหรับข้อผิดพลาดด้านความหมายหรือชนิด ในเอกสารและรายงานของเอเจนต์ ให้อ้างอิงรหัสการวินิจฉัยแทนการสรุปความล้มเหลวของคอมไพเลอร์แบบกว้าง ๆ

## คำสั่งตามภาษาของผู้อ่าน {#reader-locale}

<<<FENCE 4>>>

ผลลัพธ์ตามภาษาของผู้อ่านเป็นการเรนเดอร์แบบจำลองเชิงความหมายของคอมไพเลอร์ ไม่ใช่ชั้นแปลภาษาขณะเบราว์เซอร์ทำงาน งานด้านภาษาควรทำหลังจากแพ็กเกจตรวจสอบผ่านในรูปแบบมาตรฐานแล้ว
