# Campaign: Faber Launch And Growth

Date: 2026-07-11
Status: draft — CMO discovery selected

## Summary

Create a durable public narrative and publishing cadence for Faber. Growth
should compound from shipped language value and useful examples, not generic
announcement noise.

## Desired End State

- A memorable explanation of why Faber exists and why developers should care.
- A repeatable release/update pipeline across `faberlang.dev`, the secured X
  account, and one durable long-form channel.
- Every update connects a user problem, a visible language capability, and a
  runnable artifact without exposing private compiler internals.
- Community feedback routes into public contract issues and private
  implementation work safely.

## Audience Hypotheses To Test

1. Builders who want application code to remain readable while targeting more
   than one runtime or deployment environment.
2. Rust-adjacent developers who value explicit contracts but want a higher-level
   application language.
3. Tool and library authors interested in inspectable packages and portable
   execution lanes.

Do not target all three equally until evidence selects a primary audience.

## Campaign Path

### Stage 0 — Narrative And Category

- Owner: head-cmo with head-cpo input
- Outputs: positioning statement, category framing, audience/problem matrix,
  message hierarchy, prohibited claims, proof inventory
- Gate: every claim maps to released public evidence
- Batching: discovery-first

### Stage 1 — Content Pillars And Editorial System

- Suggested pillars: language design in public; useful examples; package
  ecosystem; release/capability updates; engineering lessons without internals
- Outputs: editorial calendar, post templates, tone guide, asset checklist,
  approval path, link/UTM convention if analytics is approved
- Batching: batch-by-default

### Stage 2 — Launch Sequence

- Pre-launch explainer, homepage, installation path, flagship example, initial
  package story, release notes, and founder narrative
- Gate: site/install/example are live and honest before promotional posting
- Batching: split-on-boundary at external publication

### Stage 3 — Ongoing Drip

- Weekly short update when there is real progress; monthly deeper technical or
  product story; release-triggered notes; occasional ecosystem spotlights
- X is the timely distribution channel; `faberlang.dev/updates` is the durable
  canonical archive. Evaluate one developer community channel after the first
  content cycle rather than spraying every network.
- Gate: no invented cadence filler and no private roadmap leakage

### Stage 4 — Feedback And Measurement

- Define meaningful signals: install success, example completion, docs search,
  package discovery, returning readers, issue quality—not follower count alone
- External analytics or mailing-list tooling requires operator approval.

## Routing Rules

- CMO owns message and channel strategy; CPO owns product truth and audience
  value; Mind converts accepted outputs into delivery work.
- No agent posts, sends email, creates accounts, or changes public profiles
  without explicit operator approval.
- Private implementation details become public only as sanitized behavior,
  release evidence, or an intentionally authored engineering story.

## Stop Conditions

Pause before first public post, paid promotion, email collection, analytics,
community moderation commitments, partnerships, or public roadmap promises.
